Drafting

So for this project we're only really going to need two classes
I put my actionlistener class in it's own folder because
since I use annonymous classes I don't like having them mixed
with other class files

First class is named Run.java which holds the main method
it also holds the following methods
Replay(): Decides whether to restart the board
or to close the window
clearBoard(): Cleans the tiktactoe board of previous x's and o's
buildBoard(): used to initiated all JComponents and set their
basic visual properties and behavior
The behavior will be handled by calling the constructor
for the Mark class which we will be getting to soon
oTurn(): Used to decide who's turn it is. Decides based
on the number of X's and O's already on the board
gamePlaying(): returns true if no win/draw condition is met
this is used to create a infinite while loop so the game is kept playing
until a win/draw condition is met

The second class is named Mark.java
It's purpose it simply to just mark down x or o depending on
what oTurn decides. False means it's not o's turn so it's x's turn
true means it is indeed o's turn
Inside the mark class there is the following methods

setOTurn(boolean turnO): used to set the variable OTurn
than we have the constructor
Mark(JButton b): it takes a given JButton and gives it an action listener
This action listener see's if the button's text is already marked
if the text is equal to "" meaning it's blank than we write to the text
an "O" if turnO is true. Else we write "X". Because if it's not clear
and it's currently no O's turn that must be it has to be marked with "X"

The reason we can't use a variable inside or a comparable is because
we cannot have an actionlistener that alters that value of an outside
variable so can't use it to change turns. That was my first attempt
and I just couldn't get that method to work so instead the oTurn was
calculated by seeing how many X's and O's are currently written down
and than calculating it from there

My Thought Process

I finished this code today and will go through my process of thinking
while creating it and will provide the full code through my github down below
Click here

The first thing I did was create the Run.java and initiate all the
JComponents. The logic behind naming my buttons b1-b9 was because
I wanted to buttons to be ordered in the same way a standard NUMPAD is
on keyboard

7 8 9
4 5 6
1 2 3

I just felt it would give the names more meaning when it came to their
positioning.
Than I just made a main method that call the constructor for the run class
than I created the Run constructor with a buildBoard since I planned on
initiating the GUI right away to make sure it's visually working

Build board was pretty simple by just initiating componnents setting
JFrame Settings than adding components to JFrame and setting JButton bounds

I than wanted to add JButton behavior but didn't want to create
an ActionListener for each button so instead I created a separate class
called Mark.java and inside that class I gave it the actionListener for
writing the JButton text.
So that way when you click on the spot on the TicTacToe board it would mark
it X or O
But at this point it would only mark X so I need to figure out how to add turns
In order to do that I need to count all the X's and O's currently on the board

So I created a method called oturn(): with two local variables
called countX and countO and checked if each JButtons text was equal
to "X" or "O" and than increment the according counting variable

After all buttons were counter for X or O I had to decide who's turn it was
Since X always starts first in a game I checked if countX was greater than countO
if the amount of x's is greater that means that x has already went so it's o's turn
if that's false than it must be X's turn to go.
But in order to change the way the ActionListener marks an X or an O I had
to set the class Mark's OTurn with a setter that I created within the Mark class

The setter would than change oTurn which would than change whether the actionListener
Marked the JButton with an X or O accordingly

OOPS
At this point I didn't plan ahead to initiate the Mark constructor with each JButton
So I simply created a Mark object and passed a JButton into it's constructor.
I could have made an actionListener for each JButton but that would have create
NINE SEPERATE ANNONYMOUS CLASSES!!!
Which is just ridiculous and bothers be because I just don't like the way
Anonymous classes look inside a directory
Also they make it annonying if you want to create a JAR exectuable file for
| your java program
Just a lot more typing and a lot more pain

So since I was running everything inside the constructor for the Run class
using a while(true) loop I need to eventually bring the game to a stop
once a game was finished
So I Created a method called gamePlaying() which checks
if the game has met a win condition or a draw condition
If neither is met than that means the game is still being played
Few minutes later after a bunch of if statements for each win condition
each returning false meaning the game has been met
If none have been met than true is returned

Now the game was working and would play for one game
After that the board would still be there but nothing would happen
You could continue to add X's and O's depending on who's turn it was last
But the game wouldn't reset or anything
So I just made a clearBoard() function to clear the board after the game was being played
and nested everything inside yet another while loop so the reset would occur after
the while(gamePlaying()) loop remained true

Now there was another problem
The game would reset before immediently after someone won
without even providing feedback for who won
So suddently the board was blank and USER would be left
wondering why the board clear

So I wanted to create a message that could be displayed
to the user to let them know who had won or if it was a draw
But I also wanted users to have the option to stop playing by
clicking a second option for NO

So now I created a new method called Replay()
looking back I should have called it Rematch
That would have made more sense but it still
gives the same idea I think
In this method it would close the application if the user
said no to having a rematch/replaying the game
but if the user said yes
than the game would clearBoard() and start all over again in a
while(true) infinite loop

for the Replay() method I created a local integer variable
called reply (I KNOW TYPO) and gave it the value of
JOptionPane.showCOnfirmDialog(null,message,"Replay",JOptionPane.YES_NO_OPTION)
so reply could only equal 1 or 0. The message would be a class String variable
inside the gamePlaying method I added a "message =" so depending the win/draw
condition this next prompt would than show the correct message like

"X WON! REMATCH?"
Killing two birds with one stone.
1. Giving the user feedback for who won
2. Giving the user the option to reset the board or
close out of the application by saying no

If the user did say NO than a System.exit(0) would be trigger
Terminating the program entirely.

If they said yes than the while(true) loop in the Run Constructor would
continue playing the game forever and forever reseting the game
for as long as the User keeps pressing Yes to rematch
